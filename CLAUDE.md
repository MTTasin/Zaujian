# Zaujain Nikah Point — Ecommerce + Configurator + AI Salesbot

Custom ecommerce for a Bangladesh business selling premium customized Nikah Nama products (books, box, pen, mirror, dupatta, thumb, frames, combos). Two apps in the repo: `backend/` (Django) and `frontend/` (Next.js). Full original spec: `zaujain_ecommerce_build_plan.md` (§15 = locked decisions). This file reflects the current built state.

## Audience — read before any customer-facing work
Non-technical people across Bangladesh incl. remote villages. Slow 2G/3G, low-end Android, low literacy.
- **Bengali only** on storefront. Terms: combo (not set), premium pen, মিরর (not আয়না).
- **Image-first, minimal text. Huge tap targets. Mobile-first. Low bandwidth** (compress + lazy-load, thumbnails not 50 full inline images).
- **Live price always visible** while configuring.
- **Admin panel is English**, technical — opposite audience.

## Tech stack
- Frontend: Next.js 16 (App Router, TS, Tailwind v4), `frontend/`. **Premium-editorial design system** (light theme): plum/rose/gold on warm cream; tokens in `globals.css` via Tailwind v4 `@theme inline` (no config file). Fonts: Noto Serif Bengali (headings, `font-display`) + Hind Siliguri (body, `font-sans`). The brand gradient (`.brand-gradient` / `.brand-gradient-text`) is now a **rose→plum accent**, not the whole identity. Shared UI primitives in `src/components/ui/`, site shell in `src/components/shell/` (SiteHeader/SiteFooter/MobileTabBar), live preview at `/ui-kit`. Charts via `recharts`.
- Backend: Django 6 + DRF, `backend/`. Single app `app`.
- DB: SQLite dev / PostgreSQL prod via `DATABASE_URL` (`dj-database-url`). Never hardcode.
- Media: Django local media folder. `next/image` is `unoptimized: true` (cPanel has no optimizer; still lazy-loads). Upload pre-sized images. **Orphaned-file cleanup**: `django-cleanup` (INSTALLED_APPS, last) auto-deletes the old file whenever an `ImageField`/`FileField` is replaced or its row deleted; the `purge_orphan_media` management command (cron) sweeps `MEDIA_ROOT` for any file no DB row references (backlog + safety net for bulk `.update()`).
- Redis (128MB) = cache/sessions only, NOT a job queue.
- Hosting: shared cPanel + Passenger. **No Docker, no background workers.** Real-time = polling, never websockets.

## Hard constraints
- **Money = `DecimalField`, never float.**
- **No job queue.** Fraud check + DeepSeek calls run synchronously with timeouts + fallbacks.
- **Snapshot prices** into cart/order at add-time; later admin edits don't mutate placed orders.

## Products & pricing
- `Product.kind` (behavior, drives configurator + pricing): `layered` (color+corner+center, e.g. book/box), `gallery` (pick one design, e.g. pen/mirror), `dupatta` (lace+lines lookup), `simple` (buy as-is or pick one design). `category` = free-text internal grouping (storefront listings carry their own `PrebuiltCombo.category`). Add unlimited product types from the panel — no code.
- Pricing engine `app/services/pricing.py`: base + option modifiers; dupatta = direct lookup; `price_bounds()` gives min/max for the "price range" step.
- **Price conflicts — customization wins (locked).** For `kind=dupatta` the option price (lace+lines) is **absolute**, so `base_price` is ignored by design: set base 1600 + an option at 1500 and everything quotes **1500**. This is consistent across surfaces — storefront shows `min_price` for any customizable product (never `base_price`), and `_shop_facts` feeds the bot the same number. `base_price` only ever renders for non-customizable products. The admin edit form warns that base_price is ignored for dupatta. (For layered/gallery/simple, options are modifiers *on top of* base_price — no conflict possible.)
- `ConfigurationImage`: a real photo for a specific color+corner+center combo — shown instead of stacked overlays when the customer's pick matches.
- **PrebuiltCombo** + `ComboImage` (multi-image) = **the catalogue model**. Every buyable listing on `/products` is one: a bundle when it links several Products, a **single item when it links one**. Tapping → Add to cart OR "Make changes" (opens wizard preloaded with the linked products). `PrebuiltCombo.category` (free text) drives the card badge and the `/products` filter — without it the card falls back to "কম্বো", which reads wrong on a single item. Plain Products are **not sold directly**; they are customizer building blocks.

## Customization rules (BUILT)
- **`Product.exclusive_group`** (admin free text, e.g. `nikahnama` on book/frame/thumb): products sharing a non-blank group can't be picked together. `/customize` **auto-swaps** (picking ফ্রেম drops বই) via the pure helper `lib/exclusive.ts` (`applyExclusive`, `exclusiveGroups`). The rule note ("বই / ফ্রেম / থাম্ব — যেকোনো একটি") is **derived from product names** — no hardcoded Bengali. Django `PrebuiltComboForm.clean()` blocks a combo mixing one group; `_shop_facts` tells the bot.
- **`Product.customize_order`** (admin) drives the picker + wizard order. The old hardcoded `ORDER` array is **deleted** (unknown categories used to sort to the top). The wizard inherits order via `sessionStorage.wizard_slugs`.
- **`ProductField`** (`product.input_fields`): admin-defined inputs asked during customization (label, placeholder, required, order) — e.g. বরের নাম, এখানে কি বসবে?. Managed per product in Admin → Products (only for non-`simple` kinds — plain items have no configurator to render them). Single-line text only.
- Answers + an always-available **optional note** ("বিশেষ নির্দেশনা") save into the existing `CartItem.config` JSON (`config["fields"]` = `[{label, value}]`, `config["note"]`) — no migration, and they inherit the price-snapshot guarantee. The **label is snapshotted next to the value**, so renaming a field never rewrites a placed order. `cart_add` **re-validates required fields server-side** (never trust the client) and caps values/note at 200 chars. `_config_display` renders them, so they appear in the cart AND the Orders admin automatically.
- Wizard has a **← পূর্ববর্তী** button (hidden on step 1). Shared configurator wiring = `useCustomerInputs(product)` hook (node/validate/payload) used by all 3 configurators.

## Customer flow
- Landing (hero + featured combos + sticky customize) → `/customize` (pick items, live **price range**) → `/customize/build` (sequential per-product configurator, confirm→next) → cart → checkout.
- **`/products` is the catalogue** — search (`?q=`) + category chips (`?category=`), listing every active PrebuiltCombo. **`/shop` and `/product/<slug>` retired**: both 308 to `/products` via `next.config.ts` redirects (query preserved). The floating customize CTA shows on **home/products/gallery only**.
- Book layered flow = 2 screens: cover (color+corner+center, live preview) then a **separate** large inside-page gallery.
- Checkout: cascading BD **Division→District→Thana** address (`frontend/src/lib/bdLocations`), WhatsApp field, email. Post-order shows "we'll call to confirm".
- **District-based delivery charge**: reduced charge for the home district (env `INSIDE_DISTRICT`, default `Chattogram` → `DELIVERY_CHARGE_INSIDE`, default 80; else `DELIVERY_CHARGE` 120). Computed **backend-authoritatively** at order create via `delivery_charge_for(district)` (`views.py`); `shop-info` returns both charges + inside district so checkout shows the right total live as district changes.

## Orders
- **`uid`** = 6-char public code; customers only ever see uid, never the numeric id. Track at `/track/[uid]` (+ `/track` to enter a code). Email at checkout + each status change includes the tracking link. The track-page **stepper starts at `confirmed`** (pending_payment step removed — most orders auto-confirm; that state still shows via the status badge + payment form).
- Cart/track thumbnail (`_resolve_preview`) resolves combo image → configured option image (color/config-photo/static/dupatta) → **falls back to the product's primary catalog photo** for as-is products (else the gift placeholder).
- Admin Orders: **guarded hard-delete** (only `pending_payment`/`cancelled`; backend enforces, others must be cancelled) + **search** (`?q=` over uid/name/phone/email).
- **`Order.cost_price`** (nullable Decimal, admin-entered per order on the detail page via the `edit` action): total cost to fulfil the order. `Order.profit` property = `subtotal − cost_price` (delivery ignored — pass-through). `null` = "not costed yet" (distinct from ৳0). Feeds dashboard **profit** = `Sum(subtotal − cost_price)` over costed, non-cancelled orders + `uncosted_count`.
- **Edit a placed order's item config** (two actions, both preserve the price-snapshot philosophy per item): `edit_config` (`orders/{id}/edit_config/`) edits only the customer's **text** — `config["fields"]` values, the note, combo line values (options/price untouched, 200-char cap); `edit_item_options` (`orders/{id}/edit_item_options/`) changes the **color/design selection** and **reprices** via `pricing.price_selection` (merges option keys, keeps text answers, recomputes `subtotal`/COD). Option edit is customizable-**product** items only; combos-as-is stay text-only.
- **Repeat customer**: same phone as a prior order → greeted (track page + email), badge in admin.
- Fraud check decides advance: aggregate success ratio **≥ `FRAUD_MIN_SUCCESS_RATIO` (70) → no advance → order goes straight to `confirmed`**; else `pending_payment` + manual bKash/Nagad step. No history / both couriers down → advance (safe default).

## Couriers
- **Fraud check** (Steadfast + Pathao): `app/services/fraud_check.py`, ported from PHP `laravel-fraud-checker/`. Merchant-login creds from env. Raw response stored on Order. Sync, per-courier timeout, fallback = require advance.
- **Consignment booking**: Steadfast official API (`Api-Key`/`Secret-Key`), only on manual admin Confirm action. `cod_amount = subtotal + delivery − advance_received`. Store consignment_id + tracking_code. On fail: don't confirm, show error.
- **Multiple consignments per order**: the primary booking stays on `Order`; extra bookings live in `ExtraConsignment` rows (FK). `orders/{id}/book_extra/` books an ADDITIONAL consignment with a per-entry **editable** payload (recipient/phone/address/COD/item description, prefilled from the order) via `create_consignment(order, invoice, overrides=)`; unique invoice `{uid}-2/-3…`; `SteadfastError` → 502, no row. Order page has an "Additional consignments" card.

## AI salesbot + human handoff  (`app/services/chatbot.py`)
- DeepSeek (`deepseek-chat`), synchronous, **streamed** (per-chunk read timeout; tolerates slow first token). Creds/config in `CHATBOT` settings from env (`DEEPSEEK_API_KEY`, model, timeout, history/token limits).
- **Persona from instructions** stored in DB (`BotConfig`), editable live at **Admin → Bot Instructions** (no restart). Seeded once from `bot_instructions.md`. Persona = **helper/guide, never a salesman** (no pressure, no "buy now", no upselling; hands off when unsure).
- **The DB is the single source of truth for products/prices** (reversed the old "prices only from instructions" rule — that left the bot with only dead `৳{price}` placeholders, so it invented numbers). `_shop_facts()` injects the **whole live catalogue** into the system prompt: every active Product (name, category, price or option range, description, `ProductSpec` rows, out-of-stock flag) + every active PrebuiltCombo (name + category, price, description = authoritative contents, linked customizable items; labelled "READY-MADE LISTINGS" since a one-item listing is not a bundle) + delivery/advance/payment. Rule: quote ONLY these, never invent; unlisted → `[HANDOFF]`.
  - `price_bounds()` returns `(0,0)` for a dupatta with no active options — `_shop_facts` **falls back to `base_price`** so the bot never quotes ৳0; a genuinely unpriced product is flagged `PRICE NOT SET` (no quote, hand off).
  - Combo item **counts are never derived** from the M2M — descriptions list items (e.g. বরমালা, স্ট্যাম্প প্যাড) that aren't separate Products, so a count would contradict the description.
  - The instruction file's `৳{price}` / `{delivery_charge}` are **placeholders nothing substitutes**; `_BEHAVIOR` tells the bot to always use LIVE SHOP DATA instead.
  - `_shop_facts` also injects a **DELIVERY/PAYMENT/CONTACT** block (delivery charge + inside-district rate, advance, COD, **bKash/Nagad numbers**, **contact phones**) and a **BUSINESS INFO** block (address, delivery time, area served, support hours, Facebook/Instagram, how-to-order + custom-order process, privacy/terms links). All from `settings.SHOP` (env-overridable). Hard rule: **never invent a phone/contact/bKash/Nagad number or address** — quote only what's fed, else `[HANDOFF]`. (Fixed a real bug where the bot hallucinated a phone number because none was provided.) Bot answers 24/7 but tells late-night customers a human confirms in the morning (support hours 5am–11pm).
- Control tags parsed out before the customer sees them: `[HANDOFF]` (→ session `waiting_admin`, bot stops) and `[GALLERY: slug]` (→ `resolve_gallery_tags` rewrites to a `{FRONTEND_URL}/gallery/<slug>` link; unknown/blank slug → the `is_bot_default` GalleryTag). `_shop_facts` lists active gallery tag slugs so the bot knows which to send. (Old `[IMAGE]`/`[ALBUM]`/ChatMedia removed.)
- Handoff = **per-chat takeover**: **Admin → Live Chats** (polling, ~4s). Admin reply → session `admin`, bot silenced **for that chat only**; back-to-bot / close. Sidebar badge + sound + browser notification on new waiting chat (poll `/api/admin/chat-unread/`).
- **Images both ways**: `ChatMessage.upload` (Pillow-capped 1600px). Customers attach images in the widget; admins attach in Live Chats. Tap → lightbox (`components/Lightbox.tsx`). Chat uploads auto-purge after 30 days via `purge_old_chat_uploads` (cron).
- Widget: floating pill on storefront (hidden on /admin), quick replies, dedupes optimistic messages by id.

## Help nudge + minimal visitor tracking  (BUILT)
- **Post-order WhatsApp**: checkout redirects to `/track/[uid]?new=1`, whose success block says "আপনার অর্ডার জমা হয়েছে!" + a tappable `wa.me` button. Number = `SHOP["WHATSAPP_NUMBER"]` (env `WHATSAPP_NUMBER`, default `01959976683`), exposed via `shop-info` (`whatsapp_number`). (The old `01818974731` was replaced by `01959976683` across footer/seo/privacy/terms/`CONTACT_PHONE`.)
- **"সাহায্য দরকার?" nudge** (`frontend/src/components/HelpNudge.tsx`, mounted in `layout.tsx`, storefront only): client-side confusion heuristic → one popup **once per session**, suppressed while chat is open (`ChatWidget` sets `body.dataset.chatOpen`). Triggers (fire on ANY, thresholds are constants at the top of the file): **idle** (`IDLE_MS` 30s no tap/scroll), **long dwell no progress** (`DWELL_MS` 60s without a `markProgress()` action), **looping** (`LOOP_LIMIT` 5 route changes across ≤2 paths), **repeat visit + empty cart**. `markProgress()`/`hasProgress()` (`lib/progress.ts`, sessionStorage) — `markProgress()` is called on real engagement (add-to-cart success in `api.ts`, combo/product card tap, customize CTA).
- **Minimal storage** (no per-visitor rows, no PII, no cleanup cron): one `DailyStat` row/day (`date` unique + `visitors`, `popups_shown`, `popups_clicked`), atomic `F()`-bumped by public `POST /api/nudge-event/` `{type: visit|shown|clicked}`. Client fires each type at most once/session (localStorage gates). Dashboard shows today's three counters (`visitors_today` / `popups_shown_today` / `popups_clicked_today`), failures swallowed so tracking never breaks the page.

## Deployment (cPanel + Passenger)  — see `DEPLOY.md`
- Frontend **`zaujain.xyz`** (Node.js App, startup `server.js`), backend **`backzaujain.mttasin.com`** (Python App, `backend/passenger_wsgi.py`), **PostgreSQL** in prod. Frontend + backend are **different registrable domains** — fine because identity is the `X-Cart-Token` header, not cookies. Just set `CORS_ALLOWED_ORIGINS=https://zaujain.xyz`, `ALLOWED_HOSTS=backzaujain.mttasin.com`, `CSRF_TRUSTED_ORIGINS=https://backzaujain.mttasin.com`, `FRONTEND_URL=https://zaujain.xyz`. (Backend can later move to `api.zaujain.xyz` with just env changes + a frontend rebuild.)
- Prod-only **security headers** auto-enable when `DEBUG=False` (settings block): `SECURE_PROXY_SSL_HEADER` (Passenger sends plain HTTP behind HTTPS), SSL redirect, secure session/CSRF cookies, HSTS, nosniff. `manage.py check --deploy` is clean with `DEBUG=False` + a real `SECRET_KEY`.
- Deploy transfer trick (slow upload line): push source zips to Google Drive, pull server-side with `python3 -m gdown "...uc?id=FILE_ID"` (fast server bandwidth) — FTP passive ports are firewalled here. - Static via **WhiteNoise** (`CompressedStaticFilesStorage`, `collectstatic`); media served by Django `re_path` when `DEBUG=False`. `CSRF_TRUSTED_ORIGINS` + `CORS_ALLOWED_ORIGINS` from env.
- **Backend secrets** → cPanel Python App env vars (or `.env`; cPanel wins), read at runtime → restart after change. **Frontend `NEXT_PUBLIC_*`** bake in at BUILD time → rebuild after change.
- Cron (no job queue): `send_pending_capi` (CAPI retry) + `purge_old_chat_uploads` (30-day chat images) + `purge_orphan_media` (monthly, unreferenced media files).
